## Highlights

- Added canonical no-silent-fallback governance through `config/fallback-governance.json` and `config/implementation-guardrails.json`, making degraded or fallback execution an explicit governed policy instead of an implicit implementation convenience.
- Added `scripts/verify/vibe-no-silent-fallback-contract-gate.ps1`, `scripts/verify/vibe-no-self-introduced-fallback-gate.ps1`, and `scripts/verify/vibe-release-truth-consistency-gate.ps1` so runtime policy, code changes, and release messaging must all agree on fallback hazards and non-authoritative degraded truth.

## Migration Notes

- Fallback is now a governed hazard surface. A degraded path must not be described or rendered as equivalent to authoritative execution.
- Contributors must not add new self-introduced fallback logic unless a requirement artifact explicitly authorizes it; otherwise the new fallback guard gate is expected to fail the release train.
- Operators reviewing a confirm or degraded route should now expect stronger warning text and `non_authoritative` framing where fallback truth is involved.
