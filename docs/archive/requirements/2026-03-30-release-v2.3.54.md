# 发布 v2.3.54 需求

**日期**: 2026-03-30
**目标**: 基于 GitHub 当前最新 `main`（`1a049f1`）和本轮已完成、已验证的非退化技术债修复成果，切出新的稳定版本 `v2.3.54`，完成受治理版本面更新、版本说明、验证收口，并在权限可用时执行 git/tag/远端发布动作。

## Intent Contract

- Goal: 发布一个能准确反映本轮已完成修复面的新稳定版本，而不是继续沿用内容失真的 `v2.3.53` 发布说明。
- Deliverable:
  - `v2.3.54` 的仓库版本面更新。
  - 与本轮修复一致的 `docs/releases/v2.3.54.md`。
  - `references/changelog.md`、`references/release-ledger.jsonl`、`docs/releases/README.md`、`dist/*/manifest.json` 的同步更新。
  - 在凭证可用时完成 commit / tag / push / GitHub release。
- Constraints:
  - 必须基于远端最新 `main`，不能基于过期主线发布。
  - 版本说明只能陈述已经在当前 worktree 中真实实现且已验证通过的修复。
  - 不能把未验证的长期规划、未来波次或假设性收益写入 release highlights。
  - 必须遵守现有 release governance、mirror sync、phase cleanup 和 node hygiene 流程。
  - 若远端发布凭证不可用，必须明确停在“本地已完成受治理发布收口”，不能谎称已远端发布。
- Acceptance Criteria:
  - `config/version-governance.json`、`SKILL.md`、`bundled/skills/vibe/SKILL.md`、`dist/*/manifest.json` 对齐到 `2.3.54 / 2026-03-30`。
  - `docs/releases/v2.3.54.md` 准确覆盖以下已完成修复面：
    - release operator closure
    - runtime contract schema baseline / docs / goldens
    - tracked outputs boundary migration
    - install-time generated nested compatibility
    - release truth / notes quality hardening
  - `release-cut.ps1 -Preview` 与 apply 结果一致，且 README / manifests / ledger / changelog 被 operator 主路径更新。
  - 发布相关 tests 与 gates 通过。
  - 阶段清理完成，node audit / cleanup 与临时文件清理有结果。
- Product Acceptance Criteria:
  - `v2.3.54` 应成为比 `v2.3.53` 更真实的稳定版本说明：不再遗漏本轮大部分修复面，也不夸大“已完全清债”。
  - 用户从 release notes、README 当前 release surface、dist manifests 和 version governance 看到的是同一版本 truth。
- Manual Spot Checks:
  - 对照 `origin/main` 与当前 worktree，确认没有漏拉最新主线。
  - 对照本轮实际通过的测试 / gates，检查 release note highlights 没有超出证据面。
  - 对照 `docs/releases/README.md` current release 与 recent releases，确认 `v2.3.54` 已替代 `v2.3.53` 成为当前 surface。
  - 若执行 git 发布，确认 tag、push 和远端可见对象对齐。
- Completion Language Policy:
  - 只有 release-cut、release note、验证和 phase cleanup 都完成后，才允许说“`v2.3.54` 已完成受治理发布收口”。
  - 只有远端 tag / release 真正创建成功后，才允许说“`v2.3.54` 已发布到 GitHub”。
- Delivery Truth Contract:
  - 本轮完成声明只覆盖已经实现并验证的修复面，不把更远期技术债路线包装为已完成。
  - 远端发布状态必须以实际 git / GitHub 操作结果为准。
- Non-goals:
  - 不在本轮继续开启新的高风险结构改造。
  - 不重写历史旧版本 release notes。
  - 不为追求整洁而回滚计划内脏改动或删除仍需保留的治理文档。
- Autonomy Mode: `interactive_governed`
- Inferred Assumptions:
  - 最合理的新版本号是 `v2.3.54`，因为当前修复面已明显超出 `v2.3.53` 说明范围。
  - 最终发布动作应先以本地受治理 version cut 和验证为主，再视远端凭证情况决定是否完成 GitHub 发布。
