# Execution Plan: Release v3.0.1 From Latest Main

## Internal Grade

`L` serial governed execution. The task is multi-step and release-sensitive, but bounded enough to avoid XL delegation.

## Wave Structure

### Wave 1

- establish isolated worktree from `origin/main`
- inspect release/tag state and release tooling
- run release-cut preview

### Wave 2

- diagnose baseline release blockers
- fix release-critical installer/nested-compatibility regressions
- re-run focused tests until green

### Wave 3

- apply the `v3.0.1` release cut
- fill release notes with truthful highlights since `v3.0.0`
- review diff

### Wave 4

- run verification commands
- commit release branch changes
- push branch, create tag, push tag, publish GitHub release

### Wave 5

- emit cleanup and delivery receipts
- summarize exact publication state

## Ownership Boundaries

- production code: installer-core nested compatibility release blocker only
- tests: fixture/runtime coverage needed to validate the release blocker fix
- release metadata: version governance, changelog, release docs, ledger, dist manifests

## Verification Commands

- `pytest -q tests/runtime_neutral/test_generated_nested_bundled.py tests/runtime_neutral/test_install_generated_nested_bundled.py`
- `pytest -q tests/integration/test_cli_installer_core_cutover.py`
- `pwsh -NoProfile -ExecutionPolicy Bypass -File scripts/governance/release-cut.ps1 -Version 3.0.1 -Updated 2026-04-09 -Preview`
- `pwsh -NoProfile -ExecutionPolicy Bypass -File scripts/governance/release-cut.ps1 -Version 3.0.1 -Updated 2026-04-09 -RunGates`

## Delivery Acceptance Plan

- do not publish until targeted tests pass
- do not state “release published” until branch push, tag push, and GitHub release creation succeed

## Completion Language Rules

- “release prepared” means code and metadata are ready locally
- “release published” means remote branch/tag plus GitHub release all exist

## Rollback Rules

- if apply or verification fails, inspect git diff and revert only the release-branch changes made in this worktree
- do not mutate the user's dirty checkout

## Phase Cleanup Expectations

- write phase receipts under `outputs/runtime/vibe-sessions/20260409T115426+0800-release-v3.0.1/`
- record whether publish completed or where it stopped
