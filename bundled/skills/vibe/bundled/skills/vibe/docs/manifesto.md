# VibeSkills Manifesto

> 我们不是在收集更多 skills。
> 我们是在为通用智能建立一层真正可治理的能力基础设施。

## 我们为什么做 VibeSkills

今天的大模型已经足够强，强到可以写代码、做研究、跑流程、连工具、调系统。

但真正阻碍它们进入稳定生产与长期协作的，不是模型能力本身，而是能力组织方式。

问题不在于没有 skills。
问题在于 skills 仍然停留在零散、脆弱、不可治理的状态：

- 用户不知道该用哪个 skill
- 团队不知道哪些 skill 值得信任
- 多个 skill 很难在复杂任务中稳定组合
- 自定义 skill 缺少规范，生态越长越乱
- 许多 agent system 看起来聪明，但行为边界、质量控制和失败回退并不清晰

我们不相信通用智能的未来会建立在 prompt glue code、野生插件和隐式魔法之上。

我们相信，大模型要真正进入真实世界，必须拥有一套可以长期演进、可验证、可回退、可组合、可协作的 skills substrate。

这就是 VibeSkills 想做的事。

## 我们反对什么

我们反对把“会调用工具”误认为“能够稳定完成任务”。

我们反对把零散 prompt、一次性技巧和不可追踪的技巧堆叠，包装成看似强大的系统。

我们反对用更多功能掩盖更深的失控，用更多 skill 数量掩盖更差的治理质量。

我们反对为了堆叠生态而堆叠生态，最后让用户自己承担选择、拼接、验证和兜底的全部成本。

我们也反对另一种常见幻觉：认为只要模型更强，这些问题自然会消失。

不会。

模型越强，外部能力越多，系统越复杂，治理问题只会更重要。

## 我们在建设什么

VibeSkills 想成为一个通用的智能大模型 skills 基座。

这意味着它不只是一个 skills list，也不只是一个 router。
它是一整套围绕能力执行建立起来的系统层：

- 它帮助模型理解任务属于什么级别、应该进入什么执行流
- 它帮助模型在多个候选 skill 中做更好的选择，而不是盲猜
- 它帮助多个 skills 在同一个任务中形成受治理的组合
- 它帮助复杂任务从单体 agent 执行，升级到 team 协作
- 它帮助整个过程拥有边界、协议、门禁、证据和回退路径

如果说模型是大脑，VibeSkills 想做的是它的执行外骨骼。

不是替代智能。
而是让智能在真实工作中更稳定地落地。

## 为什么这不是另一个 skills 仓库

一个 skills 仓库解决的是“有什么可用”。

VibeSkills 试图解决的是更难的问题：

- 什么应该在这个任务里被调用
- 什么不应该被调用
- 调用顺序如何安排
- 多个能力如何协作
- 执行质量如何验证
- 风险如何暴露
- 失败如何回退
- 长期演进如何不把系统拖入混乱

换句话说，VibeSkills 不是能力目录。
它是能力治理系统。

我们想终结的，不是 skills 本身。
我们想终结的是 skills 的混乱状态。

## 我们的技术承诺

我们不承诺“永远最强”。
我们承诺“尽可能长期地做对”。

### 1. 单一控制面优先

我们坚持以 `VCO` 作为统一控制面，而不是把多个平行编排器随意叠在一起。

新能力、新 overlay、新外部项目可以接入，但必须服从统一的治理边界、角色分工和冲突规则，而不是自带第二套控制逻辑侵入系统。

### 2. 治理先于堆功能

我们优先建设：

- 路由正确性
- 质量门禁
- memory 边界
- 冲突治理
- 可观测性
- 失败回退
- 升级与发布纪律

我们不相信“先接入、后治理”的扩张方式可以支撑长期生态。

### 3. 组合必须可解释、可验证、可回退

我们鼓励多 skill 组合，也鼓励多智能体 team。

但组合不能只是黑箱堆叠。
组合必须尽量满足：

- 为什么这样路由
- 为什么启用这个能力
- 为什么进入这个执行流
- 如何验证结果是否可靠
- 如果失败，如何降级与回退

### 4. 不把 prompt magic 冒充成系统可靠性

prompt 是能力的一部分，但不是治理本身。

我们会持续使用 prompt、overlay、语义增强和 LLM rerank 等机制来提升体验，但不会把这些手段包装成“天然可靠”的基础设施。

真正的可靠性来自协议、门禁、验证、证据与边界，而不只是更花哨的提示词。

### 5. 用户需求优先于框架自嗨

VibeSkills 的存在不是为了证明某种理论，而是为了让真实用户在真实工作里更省心、更放心、更稳定地使用大模型。

如果某项设计不能降低用户的选择成本、组合成本、风险成本或维护成本，它就不应该成为这个系统的核心。

## 我们的开源承诺

我们希望这不是一个封闭控制的体系，而是一个能吸引社区共同建设的开放基座。

### 1. 用户优先进入

我们最先想吸引的，不是框架玩家，而是正在真实使用大模型的人。

我们欢迎：

- 重度使用大模型做开发、研究、分析、自动化的个人
- 想在团队里推动 AI 落地的负责人
- 对当前 skills 生态感到疲惫和不信任的使用者

因为只有真实用户，才能不断把这个系统拉回真正有价值的方向。

### 2. 开发者围绕真实需求贡献

我们欢迎 skills 开发者、agent 框架玩家、治理工具作者、验证脚本维护者、平台集成者加入。

但我们更希望贡献是围绕真实问题生长，而不是围绕“再多接一个新玩具”生长。

### 3. 开放，不等于无边界

VibeSkills 欢迎更多能力进入生态。

但开放不意味着取消标准。
开放意味着更透明的准入边界、更清晰的角色职责、更明确的证据要求，以及更可持续的演进方式。

## 你可以如何加入

### 如果你是用户

你可以：

- 直接试用这套系统
- 提交真实任务和使用场景
- 告诉我们哪里还不稳、哪里还不够智能、哪里还不值得信任
- 帮助我们定义什么才是“可靠的 AI 执行系统”

### 如果你是开发者

你可以：

- 贡献 skills、路由策略、治理规则、验证脚本和集成能力
- 帮助降低生态重复建设与隐式冲突
- 一起把 skills 从“能用”推进到“可靠、规范、稳定、可组合”

## 这是一场什么样的建设

这不是一次性发布一个万能框架。

这是一场长期建设：

- 把离散能力组织成系统
- 把系统能力放进治理之中
- 把治理能力变成可持续演进的开源基础设施

我们不认为这件事会一夜完成。
但我们相信，通用智能要想真正进入下一阶段，这件事必须有人开始做，而且必须以开源方式做。

## 最后

VibeSkills 的野心并不温和。

我们希望做一个通用的智能大模型 skills 基座。
希望它足够通用，足够开放，足够严格，足够可靠。

我们希望未来的用户不再需要担心：

- 我现在该用哪个 skill
- 这个 skill 到底安不安全
- 多个 skill 能不能一起工作
- 这个 agent 到底会不会失控
- 这套系统是否只是一次性的演示

我们希望未来的大模型，不只是“会做很多事”。
而是“能够在一个受治理的能力系统里，长期稳定地把事情做成”。

如果你也相信这是下一代 AI 基础设施应该走的方向，欢迎来用，欢迎 Star，欢迎一起把它做出来。
